## 2. RALPLAN-DR 요약

### 2.1 Principles (설계 원칙 3~5)

1. **단일출처(Single Source of Truth)**: 158 레이어 메타는 TSV 2종에만 존재. 코드에 하드코딩 금지. NED와 동일 철학.
2. **빌드타임 정합성 보증(Compile-time Integrity)**: 카탈로그↔속성 **양방향 1:1 정합성**을 `build.rs` `assert!`로 강제 — 매핑이 깨지면 빌드 자체가 실패. 단 **절대 카운트(158)는 build.rs에서 강제하지 않고** registry 회귀테스트로 검증(레이어 추가/삭제에 빌드는 견디고, 의도치 않은 카운트 변동만 테스트가 차단).
3. **식별자 안전성(Identifier Safety)**: 데이터ID는 임의 문자열(하이픈/점/숫자선두 가능)이므로 코드젠 const 이름으로 직접 쓰지 않는다. const는 index 기반(`TWOD_ATTRS_0`)으로 생성하고 데이터ID는 문자열 필드로만 보관 — 비식별자 데이터ID로 인한 컴파일 깨짐을 원천 차단(C1).
4. **오프라인 우선(Offline-first)**: `layers`/`describe`는 API 키 없이 동작(생성된 정적 테이블 조회). 네트워크는 실제 데이터 조회(`data <ID>`)에서만.
5. **기존 자산 재사용(No New Deps)**: shapefile/encoding_rs 등 기존 의존성만. Playwright는 **1회성 스크랩 스크립트 한정**, 런타임 의존성에 추가하지 않음.
6. **최소 침습(Minimal Footprint)**: 기존 `run_data` 조회 경로는 변경하지 않음. 레지스트리 본체(`layers`/`describe`)는 그 자체로 완결되며, 조회 경로에 어떤 동작 변경도 가하지 않는다(emd-cd 완화는 본 계획서에서 제외 — M3).

### 2.2 Decision Drivers (top 3)

1. **검증된 패턴 재사용 비용 최소화** — NED-115가 이미 프로덕션 검증됨. 동일 구조 복제 시 리스크/공수 최소.
2. **데이터 수집 안정성** — 152건 미수집 속성표 스크랩이 전체 일정의 최대 리스크. 1회성·재시도·체크포인트 설계 필수.
3. **오프라인 발견성(Discoverability)** — 사용자가 "어떤 ID가 있나/속성이 뭔가"를 키 없이 즉시 알 수 있어야 함. 이것이 본 작업의 본질적 가치.

### 2.3 Viable Options (≥2)

#### Option A — 풀 코드젠 레지스트리 (NED 패턴 1:1 복제) ✅ **권장**

`twod_catalog.tsv` + `twod_attrs.tsv` → `build.rs` 코드젠 → `src/twod_registry.rs` → `data layers`/`describe`.

- **Pros**: NED와 완전 동일 구조(학습/유지보수 비용 0), 빌드타임 1:1 정합성 가드, 오프라인 동작, 단일출처, 테스트 패턴 재사용. 데이터ID는 문자열 필드로만 보관하고 코드젠 const는 index 기반(`TWOD_ATTRS_0`)이라 임의 데이터ID에도 식별자 안전(C1).
- **Cons**: `build.rs` 파싱 로직 증가, 152건 스크랩 선행 필요, 코드젠 산출물 검토 부담.

> 참고: NED는 const 이름을 camelCase endpoint_op으로 만들어도 안전했으나, 2D 데이터ID(`LT_C_UQ111`, `LP_PA_CBND_BUBUN` 등)는 식별자 보장이 없어 동일 방식이 위험. 따라서 본 계획은 const를 index 기반으로 생성한다(C1).

#### Option B — 경량 정적배열 (코드젠 없이 손작성 Rust 테이블)

TSV 없이 `src/twod_registry.rs`에 `static TWOD_LAYERS: &[...]`를 직접 손으로 작성.

- **Pros**: `build.rs` 변경 불필요, 구조 단순.
- **Cons**: 단일출처 원칙 위반(코드=데이터), 158행 수작업 = 오타·드리프트 위험, NED와 패턴 불일치(이질성), 갱신 시 코드 재작성. **드리프트 가드 부재**.

#### Option C — 온라인 describe만 (스크랩 생략, 런타임 조회)

TSV/코드젠 없이 `data describe`가 실행 시점에 상세 페이지를 매번 긁어옴.

- **Pros**: 152건 사전 스크랩 불필요, 항상 최신.
- **Cons**: **오프라인 불가**(키/네트워크 의존), 매 호출 느림·불안정, 런타임에 HTML 파싱 의존성 유입, `layers` 목록 검색 불가, NED 패턴과 완전 이질. **원칙 1·3 위반**.

> **선택: Option A.** Option B는 단일출처/드리프트 가드 상실로 탈락, Option C는 오프라인 원칙·안정성 위반으로 탈락. (1.5 ADR 참조)

### 2.4 Mode

**SHORT** (기본). NED-115라는 검증된 1:1 청사진이 존재하고 신규 아키텍처 결정이 없으므로 deliberate 불필요.
단, 152건 스크랩 안정성은 고위험 항목이므로 §7 리스크에서 pre-mortem 수준으로 별도 관리.

## 4. Guardrails

### 4.1 Must Have

- TSV 2종이 **유일한 데이터 출처** — 레이어 메타를 `.rs`에 하드코딩 금지.
- `build.rs` 정합성 가드: 카탈로그↔속성 **양방향 1:1 정합성** assert(키 집합 상호 포함). 매핑이 깨지면 빌드 실패. **절대 카운트(158) assert는 build.rs에 두지 않는다** — `src/twod_registry.rs`의 `#[cfg(test)] registry_has_158_entries` 회귀테스트가 담당(레이어 증감에 빌드는 견딤, 카운트 변동만 테스트 실패).
- 코드젠 식별자 안전성: const 이름은 **index 기반**(`TWOD_ATTRS_0`)으로만 생성하고 데이터ID는 문자열 필드로 보관. 데이터ID를 식별자로 직접 쓰지 않는다(C1).
- geom 값은 `{Polygon, Line, Point}` 3값으로 **정규화**해서만 저장/비교(MultiPolygon→Polygon, 대소문자 무시 — M2).
- `layers`/`describe`는 **API 키 없이** 동작(정적 테이블 조회).
- 크로스플랫폼: 스크랩 스크립트·코드젠 모두 경로는 `pathlib`/`Path::join` 사용, 줄끝 `\r?\n` 대응, 임시폴더는 `.` 접두 숨김.
- 스크랩 스크립트는 **재시도·체크포인트**(페이지별 분리 저장)로 세션 끊김에 견딤.

### 4.2 Must NOT Have

- **새 런타임 의존성 추가 금지**. Playwright는 스크랩 스크립트(`scripts/`)에만, Cargo 의존성 아님.
- 기존 `run_data` 조회 동작 변경 **일절 금지** — emd-cd 완화는 본 계획서 범위에서 제외(M3). 조회 경로는 한 줄도 손대지 않는다.
- 데이터ID를 코드젠 const 식별자로 직접 사용 금지 — index 기반 const만(C1).
- 레이어 메타를 코드에 손작성 금지(Option B 배제).
- 런타임 HTML 스크래핑 금지(Option C 배제).
- 158 레이어 신규 명령 158개 생성 금지 — 단일 `data <ID>` 경로 유지.

## 9. ADR (Architecture Decision Record)

- **Decision**: NED-115 코드젠 패턴을 1:1 복제하여 158 레이어용 TSV 단일출처 + `build.rs` 빌드타임 코드젠 + `data layers`/`describe` 오프라인 서브명령으로 레지스트리/카탈로그를 구현한다 (Option A).
- **Drivers**: (1) 검증된 패턴 재사용으로 리스크/공수 최소, (2) 152건 스크랩 안정성 관리, (3) 키 없는 오프라인 발견성.
- **Alternatives considered**:
  - Option B(경량 정적배열): 단일출처/드리프트 가드 상실, NED 패턴 이질 → 탈락.
  - Option C(온라인 describe만): 오프라인 불가·런타임 불안정·HTML 의존 → 탈락.
- **Why chosen**: NED-115가 프로덕션 검증된 청사진이라 동일 구조 복제 시 학습/유지보수 비용이 사실상 0이고, 빌드타임 1:1 정합성 가드로 catalog↔attrs 매핑을 컴파일러가 강제한다(절대 카운트는 회귀테스트로 분리 — 레이어 증감 견딤). 코드젠 const를 index 기반으로 두어 임의 데이터ID에도 식별자 안전하다. 오프라인 동작으로 발견성이라는 본질 가치를 충족한다.
- **CLI 표면 선택 근거(why 서브명령)**: NED는 단일 op 평면이라 플래그 확장으로 충분했으나, 2D는 "목록/검색"(`layers`)과 "단일 레이어 상세"(`describe`)라는 **두 종류의 명확히 다른 조회 의도**가 존재한다. 이를 서브명령으로 분리하면 각자 고유한 인자(`--geom/--cat/--search` vs `<데이터ID>`)를 깔끔히 가지고, 기존 `data <ID>`(실조회) 경로와도 명사적으로 구분돼 혼동이 적다. 그래서 플래그가 아닌 `layers`/`describe` 서브명령을 채택한다.
- **Consequences**:
  - (+) 단일출처·1:1 정합성 가드·오프라인·NED 일관성 확보, 갱신은 TSV만 수정.
  - (+) 빌드는 레이어 증감에 견디고(1:1만 강제), 의도치 않은 카운트 변동은 회귀테스트가 잡음(M1).
  - (+) index 기반 const로 데이터ID 식별자 안전(C1).
  - (+) 기존 `data <ID>` 조회 무변경(최소 침습, emd-cd 완화 제외 — M3).
  - (−) `build.rs` 파싱 로직 증가, 152건 선행 스크랩 필요.
- **Follow-ups**:
  - 스크랩 실패 ID 잔여분 보강 절차 문서화.
  - 속성표 미수집(빈 배열) ID 추적 및 차기 보강.
  - **(차기 별도 계획)** `--emd-cd` 강제 완화(구 부가가치 ②) — 본 계획서에서 제외. `run_data`가 attr_filter 보유 시 이미 통과하는 현 동작을 먼저 재확인하고, 실효 케이스가 분명할 때만 별도 계획으로 다룬다(M3).
  - VWorld 측 속성표 변경 시 재스크랩 트리거 정의.
