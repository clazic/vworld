## 2. RALPLAN-DR (Decision Record)

### Principles (원칙, 3~5)
1. **자기완결 배포 무결성**: skills/app 바이너리는 4개 OS 변형 모두 최신 코드(2D 158 반영)를 담아야 한다.
2. **최소 변경·과확장 금지**: 기존 3-OS 범위를 임의로 musl/arm64로 넓히지 않는다(요구 명세에 없음).
3. **문서 회귀 안전**: SKILL.md는 기존 내용을 삭제·누락 없이 전부 보존하고 신규 항목만 병합(CLAUDE.md 설정파일 수정 규칙).
4. **증거 기반 검증**: 빌드 후 `file`(아키텍처)·**호스트 무관 `strings` 게이트(4개 바이너리 모두 신규 레이어ID 임베드 확인 — 코드 최신성의 강신호)**·실행 가능한 것은 `data layers`=158으로 실증. mtime/cp는 약신호로만 취급.
5. **새 의존성 금지**: 이미 설치된 도구(rustup 타겟, docker, zig/cargo-zigbuild)만 사용. (참고: 정적 musl이 glibc ABI 문제를 원천해결하나, 배포 슬롯·검증 부담 이유로 미채택.)

### Decision Drivers (top 3)
1. **Linux 빌드의 docker 의존성** — 데몬 가동 여부가 파이프라인 성공을 좌우.
2. **코드 최신성** — 6/19 변경분이 4개 바이너리 모두에 반영되어야 함.
3. **SKILL.md 회귀 위험** — 132MB sqlite/임베드 데이터/신규 명령을 추가하되 기존 표·섹션 손실 0.

### Viable Options (≥2)

**옵션 A — build-all.sh 그대로 + docker (권장)**
- docker가 현재 UP이므로 4개 OS 변형을 한 번에 산출. 검증된 경로.
- Pros: 기존 스크립트 그대로, Linux 산출물 보장, 추가 작업 0.
- Cons: docker 데몬 의존(빌드 중 꺼지면 Linux 건너뜀), QEMU 에뮬로 Linux 빌드 느림(수분).

**옵션 B — Linux만 cargo-zigbuild로 대체 (docker 폴백)**
- macOS/Windows는 build-all.sh, Linux는 **glibc 하한 명시** `cargo zigbuild --release --target x86_64-unknown-linux-gnu.2.17` → 산출물 `target/x86_64-unknown-linux-gnu/release/vworld`를 `skills/app/vworld-linux`로 수동 복사.
- Pros: docker 불요, 빌드 빠름, glibc 하한 결정론적(2.17).
- Cons: 스크립트 우회로 수동 단계 추가, 산출 경로 직접 관리 필요.
- 비고: 하한 미지정 시 docker(rust:latest 최신 glibc)와 ABI 비결정 → 반드시 `.2.17` 명시.

**옵션 C — 빌드 범위 확장(musl/arm64 변형 추가)**
- 무효화(Invalidation): 요구 명세 "모든 os방식"은 **기존 3-OS(macOS universal/Win x64/Linux x64)** 를 의미. musl 정적링크·linux-arm64는 명세에 없고 배포 슬롯도 없어 **과확장**. 원칙 2 위반 → **채택 안 함**.

### 결정
- **A를 1차 채택**(docker UP 확인됨). 빌드 직전 `docker info`로 데몬 재확인.
- docker가 DOWN이면: (1) `orb start`/Docker Desktop 기동 후 A 재시도, 또는 (2) **B로 폴백**(Linux만 zigbuild). macOS/Windows 산출물은 docker와 무관하므로 어느 경우든 보장.

## 9. ADR (Architecture Decision Record)

- **Decision**: 기존 `scripts/build-all.sh` + docker(옵션 A)로 4개 OS 변형을 재빌드하고, docker 미가동 시 Linux만 `cargo-zigbuild`(옵션 B, **glibc 하한 `.2.17` 명시**)로 폴백한다. 빌드 범위는 기존 3-OS로 한정. 코드 최신성은 **4개 바이너리 호스트 무관 strings 게이트**(신규 레이어ID LANDINFOBASEMAP/EADIST ≥1)로 보증한다. **(C2 synthesis) 코어(바이너리+임베드 tsv)는 자기완결로 유지하고, `vworld.sqlite`(132MB)는 `--by-hjd` 고속화용 opt-in 자원으로 분리한다** — skills/app에 복사하지 않고 skills/data에 유지, SKILL.md에 `hjd-db build` 부트스트랩 절차 + `--hjd-db <path>` 사용법을 명문화한다(`--by-hjd` 기본은 sqlite 없이 역지오 폴백). SKILL.md는 `.bak` diff 회귀 게이트로 안전 병합한다.
- **Drivers**: Linux 빌드의 docker 의존성 / 코드 최신성(2D 158, strings 게이트) / SKILL.md 회귀 위험 / sqlite 132MB의 자기완결성 충돌.
- **Alternatives considered**: (B) Linux zigbuild 폴백 — docker 미가동 대비책으로 채택(2차). (C) musl/arm64 확장 — 명세 외 과확장으로 무효화. (sqlite) skills/app 동봉(자기완결 강화) vs 분리 — **분리(opt-in) 채택**: 132MB 배포 비대화 회피 + `--hjd-db` 명시 인자/역지오 폴백 구조상 동봉 불요. musl 정적링크 — glibc ABI 원천해결하나 배포 슬롯·검증 부담으로 미채택.
- **Why chosen**: A는 검증된 경로이고 docker가 현재 UP. B는 docker 리스크에 대한 결정론적 폴백 제공(glibc 하한 명시로 ABI 결정론 확보). strings 게이트는 호스트에서 실행 불가한 Linux/Windows 바이너리의 코드 최신성까지 호스트 무관으로 보증. sqlite opt-in 분리는 자기완결 원칙과 배포 경량성을 동시 충족. 모두 새 의존성 없이 설치된 도구만 사용.
- **Consequences**: 4개 OS 변형이 최신 코드로 보장(strings로 실증). docker 경로는 느리지만 안정적. zigbuild 폴백 시 산출 경로 수동 복사 1단계 추가. **sqlite는 런타임 opt-in이므로 `--by-hjd` 고속 경로 사용자는 `hjd-db build` 부트스트랩 1회 필요**(미수행 시 역지오 폴백으로 정상 동작). incremental 캐시 오염 방지를 위해 빌드 전 `cargo clean` 1회 비용.
- **Follow-ups**: open-questions 참조.
