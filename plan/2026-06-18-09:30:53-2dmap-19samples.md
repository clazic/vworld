### 1.2 샘플 공통 패턴(직접 검증 — `.vw_demo/2dsamples/*.html`)
- 전 샘플이 `vw.ol3.Map("vmap", vw.ol3.MapOptions)` + 전역 `ol`(OpenLayers 3.10.1) 사용. **현재 `render_html`(WebGL)과 완전히 다른 API 계열.**
- 좌표 두 갈래:
  - 레이어/차트/이동: **EPSG:3857 리터럴**(예: `[14086424, 4301029]`). 사용자 4326 입력 시 변환 필요.

- 벡터 등록 공식 패턴: `new ol.layer.Vector({source: new ol.source.Vector({features})})` → `vmap.addLayer(layer)`.
- **단일 폴리곤 작도 패턴(Polygon.html)**: 인라인 좌표 → `new ol.geom.Polygon([coords])` → `new ol.Feature` → `vmap.addLayer`. (범용 GeoJSON 뷰잉과 구분됨 — §2/§9 참조)
- GeoJSON 변환 가능: `new ol.format.GeoJSON().readFeatures(gj, {dataProjection:'EPSG:4326', featureProjection:'EPSG:3857'})`.
- 외부 의존: KMLLayer/Marker 샘플 일부가 jQuery·bootstrap tooltip 사용(자기완결 위해 제거/대체 대상).
- **하드코딩 경로**: KMLLayer 원본은 `/data/map/kmltest.kml`, TEXTLayer 원본은 `/data/map/4326.txt` 참조 → 사용자 입력으로 치환 필요(단계2+ 백로그 사유).

## 3. CLI 인터페이스 설계

> **명명 확정(Architect W2 반영)**: 기존 `map 2d`(WebGL 3D엔진 평면)와 신규 ol3 2D 혼동 방지를 위해 신규 키를 **`map ol`로 단일 확정**(`map flat` 병기 폐기 — CLI 공개 표면의 후속 변경비용 비대칭 때문에 사전 확정). `--help`에 명시: "**2d = 3D엔진 평면 모드 / ol = OpenLayers 데이터레이어 데모**".

```
# 기본 2D(ol3) 지도 — 신규 렌더러 (단계1)
vworld map ol --center 127.0,37.5 --zoom 11 --basemap PHOTO \
  [--controller] [--toolbar] [--zoom-control] [--overview] [--popup] -o map.html

# 마커(epsg 파라미터로 4326 그대로) (단계1)
vworld map marker --points markers.json -o markers.html
#   markers.json: [{x,y,epsg?,title,contents,iconUrl?,text?}]  (epsg 기본 EPSG:4326)

# 주제도(WMS named layer) (단계1)
vworld map theme --layers "도시지역:LT_C_UQ111,관리지역:LT_C_UQ112" -o theme.html

# 차트 오버레이 (단계1)
vworld map chart --type bar --data chart.json [--group] -o chart.html
#   chart.json: [{pos:[lon,lat], title, size?/radius?, styles:[{color,label}], values:[...]}]

# ol3 벡터 입력 (단계1 — v2.2)
vworld map ol --polygon "127.0,37.5;127.1,37.5;127.1,37.6" -o poly.html   # 인라인 폴리곤
vworld map ol --geojson features.geojson -o geojson.html                  # 범용 GeoJSON(4326 입력)
#   features.geojson: 표준 FeatureCollection(폴리곤/포인트/라인). JS측 readFeatures(featureProjection:'EPSG:3857')로 변환

# --- 단계2+ 백로그 ---
# 대량 포인트(TEXTLayer) — 파일 내용 임베드
vworld map text --file points.txt --epsg EPSG:4326 --distance 40 -o text.html
# KML(url 참조, jQuery 제거 후)
vworld map ol --kml https://example/x.kml -o kml.html

# --- 기존 Leaflet 경로(유지, 공존) ---
# 범용 GeoJSON/WFS 뷰잉 → 기존 검증된 경로(회귀 없이 유지)
vworld wfs ... --viewer   # render_wfs_viewer (Leaflet) — ol3 --geojson과 공존
```

설계 원칙:
- **데이터는 파일(`--data`/`--points`) 우선**, 소규모는 인라인 허용(`--polygon`). 차트 중첩 배열(StackedBar `values:[[...],[...]]`)은 인라인 부적합 → 파일 필수.
- 입력 좌표는 사용자 친화적 **EPSG:4326(lon,lat)** 기본. 변환 위임 전략은 §4.
- 모든 산출물에 `TOSS_STYLE` 주입(기존 헬퍼 재사용 — 단, §0.1 S1 판정 결과 반영).

### 3.1 ol3 GeoJSON 구현 메모 (v2.2)
- **데이터 주입**: GeoJSON 파일을 Rust에서 읽어 `<script>`에 변수로 주입(`ANALYSES` 임베드 패턴 답습). 절대경로 규칙(§9.4) 준수 — 외부 참조 시 `https://`만.
- **렌더 파이프라인**: `ol.format.GeoJSON().readFeatures(gj, {dataProjection:'EPSG:4326', featureProjection:'EPSG:3857'})` → `new ol.layer.Vector({source: new ol.source.Vector({features})})` → `vmap.addLayer(layer)`. 좌표는 4326 입력 → JS측 `ol.proj`가 변환(§4 일관).
- **스타일**: 토스 스타일 `#3182f6`(S1 토스 주입 결과 반영). 폴리곤/포인트/라인 각 지오메트리에 일관 적용.
- **이스케이프**: 주입 JSON 내 `</` → `<\/` (render_wfs_viewer의 `</`→`<\/` 패턴 재사용 — `</script>` 조기 종료 방지).

|------|------|-----------|------|
| 마커 | 4326 | **변환 없음** | `addMarker({epsg:'EPSG:4326'})` 런타임 내부 변환 |
| 차트 `setPosition` | 4326→3857 | **JS측 `ol.proj.transform([lon,lat],'EPSG:4326','EPSG:3857')`** | ol3 전역 `ol` 제공 확인 |
| GeoJSON(단계1 — v2.2) | 4326 | **JS측 `readFeatures(dataProjection:'EPSG:4326', featureProjection:'EPSG:3857')`** | 표준 패턴, ol3 정식 채택 |
| 초기 center/zoom | 4326→3857 | **JS측 `ol.proj.fromLonLat`** | 단일화 |
| 단일 폴리곤(단계1 — v2.2) | 4326→3857 | **JS측 `ol.proj.fromLonLat` 각 정점** | 일관성 |

결론: **모든 좌표 변환은 JS측(`ol.proj`/`fromLonLat`)으로 통일.** Rust는 데이터 직렬화·문자열 치환에만 집중. geomath.rs 메르카토르 함수 추가는 **폐기**(중복).

## 6. RALPLAN-DR 요약 (모드: SHORT)

### 원칙(Principles)
1. **자기완결 단일 바이너리** — 공식 샘플 임베드 + 문자열 치환(검증된 `ANALYSES` 패턴 답습).
2. **토스 디자인 일관 주입** — 모든 생성 HTML `</head>`에 `TOSS_STYLE`(단, ol3 DOM 비파괴 — §0.1 S1 선검증).
3. **사용자 친화 입력** — 좌표는 4326(lon,lat) 기본, 데이터는 파일 우선, 변환은 JS측 통일.
4. **회귀 안전 + 의도적 공존(v2.2)** — 기존 WebGL `render_html`·Leaflet `render_wfs_viewer` 보존. ol3는 신규 경로. **ol3=vworld 네이티브 GeoJSON 뷰어 포함 2D API 데모, Leaflet=기존·검증된 GeoJSON/WFS 뷰어.** GeoJSON 표시는 양쪽 공존(중복) — 사용자가 vworld 네이티브 일관성을 위해 명시 수용.
5. **선결 후 확장(단계화)** — 단계0 선결·스파이크 통과 후 단계1=핵심 4종, 이후 백로그 확장. 19종 요구는 단계적으로 충족.

### 결정 동인(Decision Drivers, top 3)
1. **ol3 vs WebGL API 분기** — 현 `map 2d`는 WebGL이라 ol3 샘플과 비호환 → 신규 렌더러 불가피.
2. **검증 가능성(런타임 정합)** — ol3 정합은 정적 grep만으로 불충분 → 등록 도메인/키 선결 + playwright 필수가 단계 순서를 좌우.
3. **토스 스타일 × ol3 DOM 호환** — `all:revert`/`font-family:initial`이 ol3 컨트롤·범례·popup을 깨뜨릴 위험 → 스파이크 선행이 단계1 진입 게이트.

### Viable 옵션 (≥2)
- **옵션 A — 기존 `map` 옵션 확장**: 모든 기능을 `map 2d`의 플래그로.
  - 장점: 명령 표면 최소. 단점: WebGL/ol3 모드 혼선(명명 충돌), `map`이 비대, 이질 입력 과적재.
- **옵션 B — 데이터별 새 서브명령군**(`map marker`/`chart`/`theme`/`text`):
  - 장점: 입력 스키마가 명령별로 명확, 도움말 자연스러움. 단점: 명령 수 증가.
- **옵션 C — 범용 `--layer`/`--chart` JSON 디스패치**(단일 명령 type 디스패치):
  - 장점: 확장성 최고. 단점: JSON 스키마 학습비용↑, 오류 메시지 빈약, 발견성 낮음.

### 권장(트레이드오프 결론)
**하이브리드 B+C** 채택: 토대/배경/컨트롤은 신규 키 `map ol`의 플래그(명명 충돌 회피), 이질 입력(marker/chart/theme/text)은 서브명령(B의 명확성), 각 서브명령 내부는 JSON 데이터(C의 확장성). 좌표 변환은 **JS측 ol.proj 통일**. **단계0 선결·스파이크를 게이트로 두어 검증 리스크를 단계1 이전에 차단.**

## 7. ADR (Architecture Decision Record)

- **Decision**: vworld 2D(ol3) 전용 렌더러를 신규 키 **`map ol`(단일 확정)**으로 도입한다. 단계0(선결조건 B1/B2 + 토스×ol3 스파이크 S1)을 blocking 게이트로 두고, 단계1에서 핵심 4종(`map ol`/`marker`/`chart`/`theme`) + **ol3 벡터 입력(`--polygon` 인라인 + `--geojson` 범용)**을 반영한다. TEXTLayer·KML은 단계2+ 백로그. **(v2.2 변경)** 범용 GeoJSON 뷰잉을 ol3에 정식 채택(`map ol --geojson`, `readFeatures`→`ol.layer.Vector`→`addLayer`, 공식 Polygon 코드샘플로 검증). 기존 Leaflet `render_wfs_viewer`는 유지·공존(GeoJSON 표시 2곳 중복은 사용자 명시 수용 — vworld 네이티브 일관성 우선). 좌표 변환은 JS측 `ol.proj`로 통일.
- **Drivers**: (1) ol3↔WebGL 비호환, (2) 런타임 검증 가능성(도메인/키 선결), (3) 토스 스타일×ol3 DOM 호환, (4) 자기완결·회귀안전·경계 명확.
- **Alternatives considered**:
  - A(전부 `map 2d` 플래그) — 명명 혼선·비대로 기각(신규 키 분리로 대체).
  - C(전부 범용 JSON) — 발견성·DX 저하로 단독 채택 기각, 데이터 내부 표현으로만 흡수.
  - geomath Rust 메르카토르 변환 추가 — JS측 ol.proj와 중복이라 **폐기**.
  - ol3에 `--geojson` 범용 뷰어 신규 — **(v2.2 채택)** 이전엔 Leaflet 중복으로 제외했으나, 사용자가 vworld 네이티브 일관성을 위해 ol3 GeoJSON을 명시 지시 → 중복을 수용하고 정식 채택으로 전환.
  - ol3 GeoJSON을 Leaflet로 통일(ol3 미작성) — 사용자 지시에 반하므로 기각. ol3 네이티브 뷰어 우선.
- **Why chosen**: 명령 표면 단순성·입력 스키마 명확성 균형 + 기존 `ANALYSES` 임베드 패턴 재사용으로 구현·검증 리스크 최소. 단계0 게이트로 ol3 미검증 리스크를 사전 차단. 단계화로 19종 요구와 안전 구현을 양립. **(v2.2) ol3 GeoJSON은 공식 Polygon 코드샘플의 `addLayer` 패턴으로 기술 검증되어 구현 리스크 낮음.**
- **Consequences**: 신규 ol3 렌더러 유지보수 추가; `map 2d`(WebGL)와 `map ol`(OpenLayers) 두 2D 경로 공존(`--help` 문구로 혼선 완화); **(v2.2) GeoJSON 표시가 Leaflet `render_wfs_viewer`와 ol3 `map ol --geojson` 양쪽에 존재 → 유지보수 2곳(정직히 기록). 사용자가 vworld 네이티브 일관성을 위해 의도적 중복 수용.** 단계1 착수가 B1(도메인/키) 확보에 종속(미확보 시 보류); 단계2+에서 하드코딩 경로 치환·jQuery 제거 작업 잔존.
- **Follow-ups**: **(v2.2) GeoJSON 표시 ol3 통일 옵션** — 향후 Leaflet `render_wfs_viewer`를 ol3로 일원화해 중복 제거 검토(현재는 회귀 안전 위해 공존 유지); RouteMap 영구 제외; TEXTLayer 비대화 상한 초과 시 자기완결 vs URL 외부의존 정책 확정(§9.3); 부분 구현 서브명령 롤백/정리 기준(§11); S1 판정 후 토스 스코프 최종 확정.

### 9.4 절대경로 가드 (생성 HTML 리소스)
- 생성하는 모든 HTML의 외부 리소스 URL은 **절대경로 `https://`만**. 프로토콜 상대(`//map.vworld.kr/...`)·origin 상대(`/data/map/...`) 금지.
- 2D 샘플 임베드 시 치환: `//map.vworld.kr/js/...` → `https://map.vworld.kr/js/...`, 하드코딩 `/data/map/*` → 사용자 입력 절대 URL.
- 근거: 생성물이 file://·타 서버·CDN 등 임의 환경에서 열려도 동작해야 함. (현재 3dsim/WFS/render_html은 이미 절대경로 준수.)

### 9.0 경계 재서술 (v2.2 — 사용자 지시로 이전 합의 뒤집음)
- **Leaflet `render_wfs_viewer` = 기존·검증된 GeoJSON/WFS 뷰어(유지).** 회귀 없이 그대로 둔다.
- **ol3(`map ol --geojson`) = vworld 네이티브 GeoJSON 뷰어(신규).** `ol.format.GeoJSON().readFeatures` → `ol.layer.Vector` → `vmap.addLayer` 패턴(공식 Polygon 코드샘플로 검증).
- **둘 공존(중복 수용).** 사용자가 vworld 일관성을 위해 ol3에도 GeoJSON 표시를 원함. 따라서 이전 "ol3로 범용 GeoJSON 뷰어 금지(Leaflet 전담)" 경계는 **폐기**. ol3 GeoJSON 신규 채택.
- 중복 트레이드오프는 사용자가 명시 수용(vworld 네이티브 일관성 우선). 유지보수 2곳 공존은 ADR Consequences에 정직히 기록하고, 향후 ol3 통일 옵션은 Follow-up에 둔다(§7).

### 9.1 입력 가드
- 차트: 인라인은 단일 차트만(`--values`,`--colors`). 다중/StackedBar 중첩 배열은 `--data <file>` 강제. 인라인 상한 차트 ≤8·시리즈 ≤12.
- TEXTLayer(단계2+): 인라인 임베드 ≤500포인트, 초과 시 URL/파일 참조(비대화 방지 — §9.3).
- 좌표 범위 검증: 4326 lon∈[-180,180]·lat∈[-90,90], 한국영역(124~132E,33~43N) 이탈 시 경고(3857/4326 혼동 입력 방지 — 예 `14086424`를 lon으로 착각).
- `-o` 미지정 + 대용량 임베드: stdout `html` 필드 비대화 임계 초과 시 출력 억제·파일 저장 유도.

### 9.2 자기완결 위반 사전 점검 (단계2+ 착수 선결)
- KMLLayer `/data/map/kmltest.kml`, TEXTLayer `/data/map/4326.txt` 하드코딩 → 사용자 입력 URL/데이터로 치환(잔여 상대경로 0건).
- 대상 샘플 grep: `@{apikey}` 외 내장 데이터파일·jQuery·bootstrap tooltip 의존 식별 후 제거/대체.
- B2 결과 반영: `vworldMapInit.js`의 ol 제공 확인됨 → CDN 추가 금지. jQuery는 KML/Marker만 의존 → 제거.

### 9.3 엣지케이스
- styles 길이 ≠ values 길이: 색 순환 채움(에러 대신) + 경고.
- 사용자 데이터에 `</script>`/`</` 포함(차트 라벨·속성): `render_wfs_viewer`의 `</`→`<\/` 이스케이프 재사용 필수.
- 3857 리터럴 치환 시 f64 정밀도: 충분 자릿수 직렬화(왕복 ≤1m 오차).
- 차트 위치 vs center 정합: center 미지정 시 데이터 bbox로 fit 또는 경고.
- **TEXTLayer 비대화 상한 초과 처리(명시)**: 인라인 ≤500포인트 초과 시 — **(a) 자기완결 유지하며 임베드 거부·축약 안내** 또는 **(b) URL 외부의존 허용(자기완결 예외)** 중 정책 택1을 단계2 착수 시 확정(기본값 = (a) 자기완결 우선, URL은 명시 옵트인).
