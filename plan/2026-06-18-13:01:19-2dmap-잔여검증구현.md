## 2. 구현 설계

### B1 OverviewMap — `map ol --overview` (ol2d_shell 흡수)
```js
var _ov = new vw.ol3.control.OverviewMap(vmap); _ov.site = "top-right"; _ov.draw();
```
- `run_ol`에 `if a.overview` 분기 추가. 기존 컨트롤 플래그(zoom-control/basemap-switch/popup)와 동일 패턴. **ol2d_shell 그대로 재사용**.

### B2 ToolBar — `map ol --toolbar` (ol2d_shell 흡수)
```js
var _tbOpt = {map:vmap, site:vw.ol3.SiteAlignType.TOP_LEFT, vertical:true, collapsed:false, collapsible:false};
var _tbBtns = [new vw.ol3.button.Init(vmap), new vw.ol3.button.ZoomIn(vmap), ZoomOut, DragZoomIn, DragZoomOut, Pan, Prev, Next, Full, Distance, Area];
var _tb = new vw.ol3.control.Toolbar(_tbOpt); _tb.addToolButtons(_tbBtns); vmap.addControl(_tb);
```
- `run_ol`에 `if a.toolbar` 분기. 측정(Distance/Area) 포함. **ol2d_shell 재사용**.
- 토스 스타일 충돌 점검 필요(툴바 버튼 DOM이 `#vmap` 토스 스코프 안) — S1 검증 때 PanZoomBar는 무사했으므로 동일 예상이나 playwright로 재확인.

### B3 MapController — `map controller` (신규 별도 렌더)
- **핵심 긴장(Architect)**: MapController는 `vw.ol3.Map`이 아니라 **`vw.MapController(option)`** 진입점이다. `ol2d_init_js`/`ol2d_shell`의 `vw.ol3.Map("vmap", …)` 패턴과 **호환 불가**. 또한 3D 모드는 WebGL이라 기존 `map 3d`와 기능 중복.
- **결정**: `map ol --controller` 흡수 대신 **별도 kind `map controller`** + `run_controller` 함수. ol2d_shell을 쓰되 init JS만 MapController용으로 분기(또는 전용 셸).
```js
vw.MapControllerOption = { container:"vmap", mapMode:"2d-map", basemapType:vw.ol3.BasemapType.{bm}, controlDensity:vw.ol3.DensityType.EMPTY, interactionDensity:vw.ol3.DensityType.BASIC, controlsAutoArrange:true, homePosition:vw.ol3.CameraPosition, initPosition:vw.ol3.CameraPosition };
window.mapController = new vw.MapController(vw.MapControllerOption);
```
- `<select onchange="mapController.setMode(this.value)">` 2d-map/3d-map 컨트롤 HTML 포함.
- CameraPosition은 init 전 `vw.ol3.CameraPosition.center=ol.proj.fromLonLat([lon,lat]); .zoom=z;`로 설정(ol2d 패턴 재사용 가능).
- **load 이벤트 래핑 유지**(단계1 교훈).

### B4 RouteMap — `map ol --route "lon,lat;…"` (사전 경로) + 도구
- **핵심 긴장(Critic)**: RouteMap는 본래 클릭 인터랙션 그리기 도구로 **정적 입력 데이터가 없다**(그래서 이전 ADR에서 제외). 정적 생성기가 빈 도구만 띄우면 가치가 낮다.
- **해소**: `--route "lon,lat;lon,lat;…"`로 **사전 경로 좌표를 주입**해 정적 가치 확보. 좌표가 있으면 `addRoute`로 폴리라인을 그려 보여주고, 없으면 그리기 도구만 활성화.
```js
var _rm = new vw.ol3.control.RouteMap(vmap, "route1", null, "https://map.vworld.kr/images/maps/marker.png");
// 사전 경로가 있으면 각 좌표를 addRoute로 주입(런타임 좌표는 3857)
_route.forEach(function(c){ _rm.addRoute("route1","경로","경로 지점", ol.proj.fromLonLat(c)); });
```
- 단, `addRoute` 시그니처가 좌표 객체를 요구할 수 있어(`routeMap.coordinate_` 기반) — **구현 중 실제 API 동작을 playwright로 확인하며 조정**(불확실성 명시). 사전 주입이 API상 불가하면 폴백: ol.geom.LineString 벡터로 경로를 그리고 RouteMap 도구는 빈 상태로 활성화(정적 경로 표시 보장).
- 절대경로 규칙: marker.png는 `https://map.vworld.kr/...`로.

## 3. RALPLAN-DR 요약 (모드: SHORT)

### 원칙(Principles)
1. **기존 패턴 재사용** — overview/toolbar는 `ol2d_shell`+컨트롤 플래그로 흡수(검증된 단계1 구조). 신규 렌더 경로는 꼭 필요한 controller만.
2. **정적 생성 가치 확보** — RouteMap는 사전 경로 주입(`--route`)으로 빈 도구의 무의미함을 해소.
3. **load 이벤트·vwFit·레이어 스타일 등 단계1 교훈 일관 적용**, 토스 디자인·절대경로 유지.
4. **검증 필수(게이트2)** — 신규/미검증 전 항목 playwright 렌더 확인(trusted click 포함). 회귀 0.
5. **불확실성 정직 표기** — RouteMap `addRoute` 사전주입 API는 미확인 → 폴백(LineString) 명시.

### 결정 동인(Decision Drivers, top 3)
1. **진입점 이질성** — MapController(`vw.MapController`)는 ol3 렌더러와 비호환 → 별도 경로 불가피.
2. **정적 vs 인터랙션** — RouteMap는 런타임 그리기 도구라 정적 생성 가치가 사전 경로 주입 가능 여부에 좌우.
3. **토스×ol3 DOM 호환** — 툴바/미니맵/컨트롤러 select가 `#vmap` 토스 스코프와 충돌하지 않는지 재검증 필요.

### Viable 옵션 (≥2)
- **옵션 A — 전부 `map ol` 플래그로 흡수**(overview/toolbar/controller/route 모두 `--플래그`).
  - 장점: 표면 단순. 단점: **controller는 vw.MapController라 vw.ol3.Map 기반 `run_ol`에 물리적으로 안 맞음**(init 충돌). 기각 사유.
- **옵션 B(권장) — 흡수+분리 혼합**: overview/toolbar/route는 `map ol` 플래그(ol2d_shell 재사용), controller만 별도 `map controller`.
  - 장점: 호환성·재사용 균형. 단점: 명령 1개(`map controller`) 추가.
- **옵션 C — controller·routemap 제외 유지, 검증+overview/toolbar만**.
  - 장점: 최소 작업. 단점: 사용자 "다 구현" 지시 위반. 기각.

### 권장
**옵션 B**. overview/toolbar/route = `map ol` 플래그, controller = 별도 `map controller`. RouteMap는 `--route` 사전 경로로 정적 가치 확보(폴백: LineString).

## 4. ADR (Architecture Decision Record)

- **Decision**: 미검증 4건(stackedbar/pie/group/popup)을 playwright로 검증하고, 미구현 4건을 구현한다 — overview/toolbar는 `map ol --overview/--toolbar` 플래그, RouteMap는 `map ol --route`(사전 경로 주입), MapController는 **별도 `map controller`** 명령. **이전 계획의 "RouteMap 영구 제외" ADR을 사용자 지시로 폐기**하고 정적 사전경로 형태로 포함한다.
- **Drivers**: (1) MapController 진입점 이질성, (2) RouteMap 정적 가치(사전경로), (3) 토스×ol3 DOM 호환 재검증, (4) 단계1 교훈(load/vwFit/레이어스타일) 일관.
- **Alternatives considered**:
  - A(전부 `map ol` 흡수) — controller가 vw.MapController라 `run_ol`(vw.ol3.Map)과 init 충돌로 기각.
  - C(controller/routemap 제외 유지) — "다 구현" 지시 위반으로 기각.
  - RouteMap를 빈 그리기 도구로만 — 정적 가치 낮아 `--route` 사전주입으로 보완.
- **Why chosen**: 검증된 `ol2d_shell` 최대 재사용 + 진입점이 다른 controller만 격리 → 구현·검증 리스크 최소. RouteMap 사전경로로 정적 산출물 의미 확보.
- **Consequences**: `map ol` 플래그 3종(--overview/--toolbar/--route) 추가 + 신규 `map controller` 명령. MapController는 2D/3D 전환이라 기존 `map 2d/3d`와 일부 기능 중복(문서로 구분). RouteMap `addRoute` 사전주입 API 미확인 → 구현 중 확정(폴백 준비).
- **Follow-ups**: addRoute 사전주입 가능 여부 확정 후 폴백 채택 여부 기록. 토스 스코프가 툴바/select를 깨면 §0.1 S1식 대응(분리/축소).
